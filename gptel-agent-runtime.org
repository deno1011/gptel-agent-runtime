#+TITLE: gptel-agent-runtime
#+PROPERTY: header-args:emacs-lisp :tangle yes :lexical t

Master entrypoint for =deno1011/gptel-agent-runtime=. After the
2026-05-27 monolith split this file is a thin loader: it carries the
package header, declares the host =my/data-dir= fallback, requires the
=gar-*= modules in dependency order, runs the small tail-of-load wiring
(Ollama-active-model selection, Response Executor activation, and global
=gptel-tools= setup), and =(provide)='s the package feature.

* Architecture Summary

** Runtime Flow

A user gptel-send (or an interactive =M-x gptel-agent-runtime-start=)
travels through this layered stack:

1. =gar-substrate= advances the OpenClaw tick and emits a user-request
   event through the pump.
2. =gar-agents= classifies the request (chat router score, model router
   profile, organization unit, agent role, matching skills/playbooks).
3. =gar-policy= evaluates the resulting plan steps -- capability gate,
   policy alist, quarantine pre-flight (=gar-quarantine=), and the
   Advocatus Diaboli skeptic (=gar-skeptic=) -- and emits a
   =policy-decision= event observed by =gar-mission-control=.
4. =gar-loop= dispatches each allowed step to a worker, observes the
   result, reflects, and either continues or finalizes.
5. =gar-tools= runs the native gptel tools each step delegates to.
6. =gar-memory= persists sessions, scores playbooks, retrieves prior
   context, synthesizes candidate playbooks on idle ticks.
7. =gar-executor= watches the gptel response and runs Babel / AUTORUN /
   file-output blocks for inline rendering.
8. =gar-context= attaches workspace context and image/web helpers.
9. =gar-directives= picks the appropriate gptel directive for the
   active backend.
10. =gar-backend= provides the Ollama runtime utilities; the consuming
    setup file is responsible for the actual backend constructors.
11. =gar-core= is the layer underneath everything else: defgroup,
    defcustoms, every cl-defstruct, base helpers.

** Functionality catalog

The runtime currently gives you these user-visible capability families:

- /Normal gptel integration/: backend/model selection, local Ollama helper
  startup, runtime-specific directives, global tool activation, image/web
  context helpers, and response execution for Org/Babel/file-output blocks.
- /Autonomous task execution/: a planner creates a session and plan, workers
  execute allowed steps, reflection decides whether to continue, and the
  session finalizes into memory and trajectory records.
- /Routing/: chat-to-swarm routing decides when ordinary chat should become
  an autonomous session; agent/organization routing chooses a role and process;
  model routing can switch to a configured backend/model profile for harder or
  riskier prompts.
- /Tools/: a registry wraps native gptel tools, high-risk argument schemas,
  Org/file/buffer/web helpers, raw local-model tool shims, and a reviewable
  tool-invention pipeline.
- /Guardrails/: capability checks, per-tool policy, protected path checks,
  blocked shell/placeholders, quarantine of untrusted evidence, skeptic review,
  JSON-like argument validation, post-execution verification, completeness
  checks, retry preparation, and prompt-injection canaries.
- /Memory and learning/: session memory, embedding cache, playbook scoring,
  novelty detection, strategy synthesis, trajectory storage, SQLite/FTS/vector
  retrieval, failure analytics, playbook refinement, A/B experiments, markdown
  skills, and trajectory-to-skill promotion with trust/transfer-trust.
- /Observability/: swarm trace, event pump view, evidence trace, mission
  control, tool-policy editor, failure report, skill promotion review, and
  status/command-center entry points.

** Main execution paths

The function reference sections in each =gar-*.org= list every function in
that file. The flows below explain how those functions compose at runtime.

*** Package load

1. =gptel-agent-runtime.org= requires modules in dependency order.
2. =gar-core= defines all shared options, structs, and constructor helpers.
3. =gar-substrate= installs the event/tick/evidence substrate that later
   modules subscribe to.
4. Guardrail, memory, tool, backend, directive, context, executor, agent, and
   loop modules load their registries and command definitions.
5. Tail-of-load wiring forwards =my/data-dir=, activates the Response
   Executor, selects the default local model if an Ollama backend exists, and
   enables global gptel tools once =gptel= itself is loaded.

*** Ordinary gptel send

1. =gar-agents--inject-context= may add workspace context around =gptel-send=.
2. =gptel-agent-runtime-sync-directive-for-current-runtime= keeps the active
   directive matched to the current backend/model.
3. =gar-response-executor-mode= watches the assistant response.
4. When the response contains executable blocks or file-output markers,
   =gar-executor= validates, runs, renders, and records outputs.

*** Chat routed to autonomous runtime

1. =gptel-agent-runtime--maybe-route-chat-to-swarm= receives the latest chat
   text and asks =gptel-agent-runtime-classify-chat-request= whether to route.
2. =gptel-agent-runtime-route-task= matches organization units, agents, skills,
   and playbooks; =gptel-agent-runtime-apply-model-route= may switch model.
3. =gptel-agent-runtime-start= creates a session and enters the loop.
4. =gar-loop= observes workspace/memory, asks the planner for steps, evaluates
   each step through =gar-policy=, dispatches allowed tool work, verifies
   results, reflects, and either retries, continues, or finalizes.
5. Finalization emits events consumed by trajectory, memory, failure
   analytics, refinement, experiments, and skill-promotion subscribers.

*** Tool execution and safety

1. A plan step names a tool and arguments.
2. =gptel-agent-runtime-policy-evaluate-step= checks capability allowlists,
   argument schema, protected paths, command/path placeholders, quarantine
   conflicts, per-tool policy, and skeptic verdicts.
3. Allowed steps run through worker dispatchers in =gar-loop= and native tool
   functions registered by =gar-tools=.
4. =gptel-agent-runtime-verify-action-result= checks the observed result; when
   configured, completeness verification compares requested counts against
   prior tool output.
5. Failures become verifier verdicts, failure analytics rows, retry hints, and
   eventually playbook-refinement evidence.

*** Learning loop

1. Successful and failed sessions become trajectories in =gar-trajectory= and
   optional SQLite rows in =gar-memory-sqlite=.
2. =gar-memory= ranks matching playbooks, detects novelty, and can synthesize
   candidate strategies.
3. =gar-playbook-refine= watches consistent failures and writes candidate
   refinements.
4. =gar-playbook-experiment= can A/B test refined playbooks against originals.
5. =gar-skill-promote= watches successful trajectories, writes markdown skill
   candidates, tracks user approval, promotes trusted repeated patterns, and
   transfer-approves similar future candidates.

** Load order

The master =(require)='s modules in this order, which respects the
dependency DAG: =gar-core= -> =gar-substrate= -> =gar-quarantine= ->
=gar-skeptic= -> =gar-skeptic-model= -> =gar-validator= -> =gar-policy=
-> =gar-mission-control= -> =gar-canaries= -> =gar-memory= -> =gar-tools=
-> =gar-backend= -> =gar-directives= -> =gar-context= -> =gar-executor=
-> =gar-agents= -> =gar-loop=.

Modules forward-declare cross-module callees with =declare-function= and
read defcustoms from =gar-core= via =defvar= forward declarations so
isolated byte-compilation stays warning-free.

** Development rule

- Edit the matching =gar-*.org= for the area you are changing.
- Tangle that =.org= and rerun
  =emacs -Q --batch -l scripts/refresh-backlinks.el -f gar-refresh-backlinks-and-exit=
  so each module's =Required by= footer reflects the current
  =(require 'gar-...)= edges.
- Validate the tangled =.el= with =check-parens= and a batch load smoke
  test before committing.
- Keep =main= and =stable= synced unless =stable= intentionally diverges.

** Complexity map

The split is not mostly ceremony: the runtime now contains several
independent subsystems that only make sense when they can evolve in
separate modules.

- =gar-core=, =gar-substrate=, =gar-policy=, =gar-tools=, =gar-loop=, and
  =gar-agents= are the core runtime. Without them there is no typed state,
  event pump, safety gate, tool registry, execution loop, or routing layer.
- =gar-backend=, =gar-directives=, =gar-context=, and =gar-executor= are
  gptel integration helpers. These are user-visible and could stay enabled
  even if the autonomous runtime is off.
- =gar-memory=, =gar-trajectory=, =gar-memory-sqlite=,
  =gar-playbook-refine=, =gar-playbook-experiment=, =gar-skills-md=,
  =gar-skill-promote=, and =gar-failure-analytics= form the learning
  layer. This is where most optional complexity lives: it exists to remember
  what worked, find similar past runs, refine playbooks, promote durable
  skills, and explain recurring failures.
- =gar-quarantine=, =gar-skeptic=, =gar-skeptic-model=, =gar-verifier=,
  =gar-validator=, and =gar-canaries= are guardrails. They deliberately add
  lines because they make model/tool execution inspectable instead of
  trusting raw generated text.
- =gar-mission-control= is observability and UI: it makes the above state
  visible in Emacs buffers.

If the goal is fewer lines for daily use, the safest simplification is not
to merge the files. A better path is to make optional layers lazy or
feature-gated: keep the core/runtime/gptel integration loaded, and load the
learning, experiment, SQLite, model-skeptic, canary, and mission-control
layers only when their defcustoms or commands are used.

#+begin_src emacs-lisp
;;; gptel-agent-runtime.el --- Emacs-native agent runtime on top of gptel -*- lexical-binding: t; -*-

;; Copyright (C) 2026 Denis Butic

;; Author: Denis Butic
;; Maintainer: Denis Butic
;; URL: https://github.com/deno1011/gptel-agent-runtime
;; Version: 0.1.0
;; Package-Requires: ((emacs "29.1") (gptel "0.9.9"))
;; Keywords: convenience, tools, ai, gptel

;;; Commentary:

;; Thin master entrypoint for the multi-file gptel-agent-runtime
;; package. After the 2026-05-27 monolith split, every concern lives in
;; a dedicated `gar-*' module. This file pulls them in in dependency
;; order and runs the small tail-of-load wiring that needs to happen
;; only after every module has loaded.
;;
;; Host configuration MAY define `my/data-dir' before requiring this
;; package to override the default data directory. When `my/data-dir'
;; is bound at require time, this file forwards its value into the
;; package's own `gptel-agent-runtime-data-directory' defcustom (see
;; gar-core); when it is not bound, the defcustom's default
;; (`user-emacs-directory') is used. The package itself no longer bakes
;; in a personal path like `~/emacs/'.

;;; Code:

(require 'cl-lib)
(require 'json)
(require 'subr-x)

;; Backwards-compat shim: some host configs define `my/data-dir' before
;; requiring this package, and some modules (notably gar-directives) read
;; the variable at LOAD TIME to bake the value into directive prompts. To
;; keep both paths working: defvar with `user-emacs-directory' as the
;; default so a fresh install sees a sensible non-nil value, but let any
;; pre-bound host value win (defvar is a no-op when the variable is
;; already bound). The package's canonical setting is
;; `gptel-agent-runtime-data-directory' in gar-core; the tail-of-load
;; wiring below forwards whatever value `my/data-dir' ended up with into
;; that defcustom so the rest of the runtime sees a single source of
;; truth.
(defvar my/data-dir user-emacs-directory
  "Personal data root, optionally pre-bound by the host config.
When the host config sets this before requiring the package, that value
is forwarded into `gptel-agent-runtime-data-directory' at the tail of
the master load sequence. New setups should customise
`gptel-agent-runtime-data-directory' directly.")

;; Module load order (respects the dependency DAG).
(require 'gar-core)        ; defgroup + defcustoms + all cl-defstructs + base helpers
(require 'gar-substrate)   ; tick, event pump, evidence, versioned state
(require 'gar-quarantine)  ; per-source quarantine + promote + pre-flight conflict check
(require 'gar-skeptic)     ; Advocatus Diaboli core (rule-based + dispatcher)
(require 'gar-skeptic-model) ; model-based skeptic verdict path
(require 'gar-verifier)    ; post-execution verifier + auto-retry signal
(require 'gar-validator)   ; JSON-Schema-inspired tool-arg validator
(require 'gar-policy)      ; policy broker, capability gate, presets, context wrappers
(require 'gar-mission-control) ; unified runtime dashboard
(require 'gar-canaries)    ; prompt-injection canary suite (wraps gar-policy's untrusted-context)
(require 'gar-memory)      ; sessions, embedding cache, novelty, synthesis, hypothesis-test
(require 'gar-trajectory)  ; per-goal trajectory storage (substrate for learning loop)
(require 'gar-playbook-refine) ; auto-refine playbooks on consistent failure
(require 'gar-playbook-experiment) ; A/B test refined playbooks against originals
(require 'gar-memory-sqlite) ; SQLite/FTS5 + vector trajectory retrieval (optional)
(require 'gar-skills-md)   ; markdown skill format + load + emit refinement candidates as md
(require 'gar-skill-promote) ; trajectory-to-skill promotion (success side of the learning loop)
(require 'gar-failure-analytics) ; visible failure-pattern aggregation for mission control
(require 'gar-tools)       ; tool registry, native gptel tools, tool-invention pipeline
(require 'gar-backend)     ; Ollama runtime utilities + model-id normalization
(require 'gar-directives)  ; emacs-local-assistant / planner / assistant directives
(require 'gar-context)     ; image capture + web fetch helpers
(require 'gar-executor)    ; Response Executor (formerly claude-executor-*; aliases kept)
(require 'gar-agents)      ; agent/skill/org-unit/playbook registries + chat + model router
(require 'gar-loop)        ; autonomous execution loop + worker dispatcher

;; If the host config set `my/data-dir' before requiring us, forward
;; that value into the package's own data-directory defcustom so the
;; rest of the runtime sees a single source of truth. Setting only
;; happens when the host explicitly opted in -- the defcustom's default
;; (`user-emacs-directory') wins on a fresh install.
(when (and (boundp 'my/data-dir)
           (stringp my/data-dir)
           (boundp 'gptel-agent-runtime-data-directory))
  (setq gptel-agent-runtime-data-directory
        (file-name-as-directory my/data-dir)))

;; Tail-of-load wiring. These steps need to run only AFTER every module
;; has finished loading because they consult cross-module helpers.

;; Activate the Response Executor (defined in gar-executor) now that
;; every module has loaded. The function is `gar-response-executor-mode';
;; the legacy `claude-executor-mode' name remains available via
;; `defalias' for any user config that still binds it.
(when (fboundp 'gar-response-executor-mode)
  (gar-response-executor-mode 1))

;; Select the local model only after directives, tools, and
;; compatibility helpers have been defined.
(when (and (boundp 'gptel-agent-runtime-ollama-backend)
           gptel-agent-runtime-ollama-backend)
  (gptel-agent-runtime-use-default-local-model))

;; Finish wiring gar-tools' global tool list now that gptel-agent-runtime-tools-all
;; is defined (it lives in gar-core which loads before gar-tools, so
;; this with-eval-after-load runs only when gptel itself is loaded).
(with-eval-after-load 'gptel
  (when (and (fboundp 'gptel-agent-runtime-tools-all)
             (boundp 'gptel-use-tools))
    (setq gptel-use-tools t
          gptel-tools (gptel-agent-runtime-tools-all))))

(provide 'gptel-agent-runtime)

;;; gptel-agent-runtime.el ends here
#+end_src
